# ISTQB CTAL-TA Key Terms Glossary

## A

**accessibility**
The degree to which a component or system can be used by people with the widest range of characteristics and capabilities to achieve a specified goal in a specified context of use.
*Synonyms: user assistance*

**adaptability**
The degree to which a component or system can be adapted to various or evolving hardware, software, or other operational or usage environments.
*Reference: After ISO 25010*

**ad hoc reviewing** (ad hoc review)
A review technique performed informally without a structured process.
*Reference: ISO 20246*

## B

**behavior-based test technique**
A black-box test technique that derives test cases from a model of the dynamic behavior of a test item.
*See also: CRUD testing, state transition testing, scenario-based testing*

**boundary value**
A minimum or maximum value of a range within an equivalence partition.

## C

**checklist-based reviewing**
A review technique in which a predefined set of questions or criteria are used to verify a work product.
*Reference: ISO 20246*

**checklist-based testing**
An experience-based test technique in which a predefined list of conditions is used for test analysis, design, and execution.

**combinatorial testing**
A black-box test technique in which test cases are designed to exercise specific combinations of values of several parameters.

**compatibility**
The degree to which a component or system can exchange information with other components or systems and/or perform its required functions while sharing the same hardware or software environment.

**crowd testing**
A test approach in which testing is distributed to a large group of testers.

**CRUD testing**
A black-box test technique in which test conditions are operations of the types create, read, update, and delete for an entity.

## D

**data-based test technique**
A black-box test technique that derives test cases from the data domain of a test item.
*See also: domain testing, combinatorial testing, random testing*

**decision table testing**
A black-box test technique in which test cases are designed to exercise the combinations of conditions and the resulting actions shown in a decision table.

**defect prevention**
The activity of identifying and removing causes of defects early in the software development lifecycle.

**domain testing**
A black-box test technique in which test conditions are selected from the domain of possible values of the variables of a test item.

**DRE (Defect Removal Efficiency)**
The ratio of defects removed before the release to the total number of defects identified. Measures overall effectiveness of defect prevention and detection practices.

## E

**equivalence partition** (equivalence class)
A subset of the value domain of a variable within a component or system in which all values are expected to be treated the same.

**experience-based testing**
Testing based on the tester's experience, knowledge, and intuition.

## F

**functional appropriateness**
The degree to which the functions of a component or system facilitate the accomplishment of specified tasks and objectives.
*Reference: ISO 25010*

**functional completeness**
The degree to which the set of functions covers all the specified tasks and user objectives.
*Reference: ISO 25010*

**functional correctness**
The degree to which a component or system provides the correct results with the needed degree of precision.
*Reference: ISO 25010*

**functional suitability**
The degree to which a component or system provides functions that meet stated and implied needs when used under specified conditions.
*Reference: ISO 25010*

**functional testing**
Testing conducted to evaluate the compliance of a component or system with functional requirements.

## H

**high-level test case** (abstract test case, logical test case)
A test case without concrete values for input data and expected results. Typically used as a basis for the derivation of concrete test cases.

## I

**impact analysis**
The identification of all work products affected by a change, including an estimate of the resources needed to accomplish the change.

**installability**
The degree of effectiveness and efficiency with which a component or system can be successfully installed or uninstalled in a specified environment.
*Reference: ISO 25010*

**interaction capability**
The degree to which a component or system enables easy operation and control of the users.
*Reference: ISO 25010*

**interoperability**
The degree to which two or more components or systems can exchange information and use the information that has been exchanged.
*Reference: ISO 25010*

## K

**keyword** (action word)
A label used in keyword-driven testing to represent an action or verification step.

**keyword-driven testing**
A scripted test technique in which test scripts consist of keywords that represent test actions and verification steps.
*See also: action word testing*

## L

**low-level test case** (concrete test case, physical test case)
A test case with concrete values for input data and expected results. Derived from high-level test cases.

## M

**metamorphic relation**
A description of how a change to the input(s) of a test item is reflected in a change to the output(s) of the test item.

**metamorphic testing**
A test technique in which test cases are derived based on metamorphic relations, applying transformations to existing test cases to create follow-up test cases.

## P

**PCE (Phase Containment Effectiveness)**
The ratio of defects introduced and removed in the same phase to the total number of defects introduced in that phase. Measures how effectively defects are contained within the phase they are introduced.

**perspective-based reading** (PBR)
A review technique in which a reviewer simulates the role of a stakeholder to read a work product and derive the information that the stakeholder would use.

**phase containment**
The detection and removal of defects in the same phase in which they were introduced.

**product risk**
A risk that is related to the quality of the test object.

## R

**random testing**
A black-box test technique in which test cases are designed using pseudo-random values from the value domain of a test item.

**regression testing**
A type of change-related testing to detect whether defects have been introduced or uncovered in unchanged areas of the software.

**review technique**
A specific approach used by a reviewer to review a work product, such as ad hoc reviewing, checklist-based reviewing, scenario-based reviewing, role-based reviewing, or perspective-based reading.

**risk analysis**
The overall process of risk identification and risk assessment.

**risk assessment**
The process to examine identified risks and determine the risk level.

**risk-based testing**
A test approach in which the management, selection, prioritization, and use of testing activities and resources are based on corresponding risk types and risk levels.

**risk control**
The overall process of risk mitigation and risk monitoring.

**risk identification**
The process of finding, recognizing, and describing risks.

**risk mitigation**
Actions taken to reduce the probability of a risk occurring or to reduce the impact of a risk if it does occur.

**risk monitoring**
The activity of tracking identified risks and taking action if risk levels change.

**root cause analysis** (RCA)
An analysis technique aimed at identifying the root causes of defects. By addressing root causes, organizations can prevent future recurrence of similar defects.

**rule-based test technique**
A black-box test technique that derives test cases from rules that describe the stateless behavior of a test item.
*See also: decision table testing, metamorphic testing*

## S

**scenario-based reviewing**
A review technique in which a reviewer uses a use case or scenario to review a work product.

**scenario-based testing**
An experience-based test technique in which test cases are derived from scenarios. Also used as a black-box technique deriving test cases from scenario models.

**session-based testing**
An exploratory testing approach that uses time-boxed sessions, each guided by a test charter.

**state transition testing**
A black-box test technique in which test cases are designed to exercise elements of a state transition model.

## T

**test analyst**
A person responsible for test analysis, test design, test implementation, and test execution, focusing on non-technical aspects of software.

**test basis**
The body of knowledge used as the basis for test analysis and test design.
*Reference: After TMap*

**test case**
A set of preconditions, inputs, actions (where applicable), expected results and postconditions, developed based on test conditions.

**test charter** (charter)
Documentation of a test objective for a test session.
*See also: exploratory testing*

**test condition** (test situation, test requirement)
A testable aspect of a component or system that is intended to be tested.
*Reference: After ISO 29119-1*

**test data**
Data needed for test execution.

**test design**
The activity that derives and specifies test cases from test conditions.

**test environment** (test bed, test rig)
An environment containing hardware, instrumentation, simulators, software tools, and other support elements needed to perform a test.
*Reference: ISO 24765*

**test execution**
The activity that runs a test on a component or system producing actual results.

**test implementation**
The activity that prepares the testware needed for test execution.

**test oracle**
A source to determine an expected result to compare with the actual result of the system under test.

**test script**
The instructions for the execution of a test, either manual or automated.

**testware**
Work products produced during the test process for use in planning, designing, executing, evaluating, and reporting on testing.

**traceability**
The degree to which a relationship can be established between two or more work products.

## U

**usability**
The degree to which a component or system can be used by specified users to achieve specified goals with effectiveness, efficiency, and satisfaction in a specified context of use.

**user experience** (UX)
A person's perceptions and responses that result from the use or anticipated use of a product, system, or service.
